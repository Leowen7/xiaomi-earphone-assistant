# AJ15XIAOMI-9：小米耳机智能选购与客服助手 MVP 范围说明

## 1. 项目定位

本项目面向小米和 REDMI 耳机产品，以结构化产品数据处理为核心，建设一个同时支持“产品参数对比、个性化选购推荐、产品使用问答”的智能助手。

第一阶段以 8 款耳机为样本，先验证完整业务链路与技术架构，不追求覆盖全部小米产品。

## 2. 第一阶段目标

第一阶段需要完整跑通以下三条业务链路：

1. 产品参数对比  
   用户选择两款耳机，系统读取结构化产品数据，对齐公共参数，展示价格、佩戴方式、重量、续航、降噪、防水、连接能力等差异，并输出各自优势。

2. 个性化选购推荐  
   用户提供预算、使用场景、必须条件和功能偏好，系统先执行硬条件筛选，再按场景权重进行标准化评分，返回前 3 款产品及推荐理由和不足。

3. 说明书 RAG 问答  
   用户选择产品并提出配对、充电、重置、降噪切换或故障处理问题，系统从说明书 FAISS 知识库中检索相关片段，大模型仅依据检索内容组织回答，并展示文件名、页码和原文片段。

## 3. 第一阶段产品范围

第一阶段固定选择 8 款小米或 REDMI 耳机，产品编号为 EAR001—EAR008。产品应覆盖：

- 2 款入门级产品；
- 3 款中端产品；
- 2 款高端或旗舰降噪产品；
- 1 款开放式产品。

具体名单见 `data/earphone_list.xlsx`。

## 4. 第一阶段功能范围

### 4.1 产品数据管理

- 建立统一的产品数据字段；
- 记录每项参数的官方来源；
- 统一价格、重量和续航单位；
- 统一布尔值、枚举值和缺失值；
- 生成可由后端读取的 JSON 数据；
- 输出数据清洗与质量报告。

### 4.2 产品参数对比

- 支持选择任意两款耳机；
- 对齐公共参数；
- 计算数值差异；
- 比较降噪、防水、双设备和低延迟等功能；
- 缺失参数显示“暂无官方数据”；
- 大模型不得自行补充产品参数。

### 4.3 个性化推荐

第一阶段支持以下输入：

- 预算上限；
- 日常、通勤、运动、游戏场景；
- 是否必须主动降噪；
- 是否必须防水；
- 是否必须低延迟；
- 是否重视轻便；
- 是否重视长续航；
- 是否排除入耳式。

推荐逻辑采用：

`硬条件筛选 → 指标标准化 → 场景权重评分 → 排序 → 返回前 3 款 → 生成解释`

产品排名必须由程序计算，大模型仅负责解释程序结果。

### 4.4 说明书问答

- 支持按产品 ID 限定检索范围；
- 每次检索返回 3—5 个说明书片段；
- 回答包含来源文件、页码和原文；
- 无足够依据时明确提示，不得编造答案。

### 4.5 前端与后端

前端采用 HTML、CSS、JavaScript，后端采用 Python Flask。

第一阶段核心接口：

- `GET /api/products`
- `POST /api/compare`
- `POST /api/recommend`
- `POST /api/chat`

## 5. 第一阶段不实现

以下内容暂不纳入本 Sprint：

- 手机、平板、手环、路由器等其他产品类别；
- 用户注册、登录和权限系统；
- 商品购买与支付；
- 实时电商价格抓取；
- 图片识别和语音输入；
- 模型微调；
- 基于用户行为的协同过滤推荐；
- 多智能体；
- 多语言客服；
- 大规模自动爬虫；
- 复杂部署和高并发优化。

## 6. 角色与职责

- 组长：范围、数据规范、接口规范、Jira、GitHub、Flask 集成、测试与演示；
- 组员 A：产品参数、官方来源、重点说明书；
- 组员 B：数据清洗、产品对比、推荐规则和推荐算法；
- 组员 C：说明书解析、FAISS 检索、前端页面。

## 7. 关键数据文件

- `data/earphone_list.xlsx`：第一阶段产品名单；
- `data/data_dictionary.xlsx`：字段定义、单位和缺失值规则；
- `data/raw/product_data_raw.xlsx`：原始产品参数；
- `data/processed/product_data_clean.json`：清洗后的结构化产品数据；
- `data/processed/chunks.jsonl`：说明书文本块；
- `vector_store/index.faiss`：向量索引；
- `vector_store/metadata.jsonl`：向量元数据。

## 8. MVP 验收标准

- 8 款产品拥有可追溯的结构化数据；
- 至少 4 款产品具有有效说明书知识；
- 支持任意两款产品对比；
- 支持预算、场景和偏好的推荐；
- 返回前 3 款产品、得分、理由和不足；
- 支持说明书问答及来源展示；
- 前端三个功能入口可以使用；
- 4 个核心后端接口可以调用；
- 综合测试不少于 20 组；
- GitHub `main` 保存可运行的稳定版本。

## 9. 变更管理

本文件合并到 `main` 后视为第一阶段范围冻结。新增产品类别、修改公共字段、修改接口请求或返回格式时，必须先由组长确认，并在 Jira 和本文件中记录变更。
